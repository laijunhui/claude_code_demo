# 多Agent代码自动评审工具 - 产品需求文档（PRD）

**版本：** V1.0（MVP）
**目标用户：** 中小技术团队（3-15人）
**文档状态：** 初稿

---

## 1. 产品概述

### 1.1 产品定位

为中小团队提供**轻量化、私有化、高性价比**的多Agent代码自动评审工具，通过自动化评审提升代码质量、降低人工review负担。

### 1.2 核心价值

- **快**：PR提交后分钟内完成评审，无需等待
- **准**：多维度检查，覆盖语法、风格、安全、性能
- **省**：Docker一键部署，降低使用门槛
- **私有**：数据本地处理，满足合规要求

### 1.3 目标用户

| 用户 | 核心需求 |
|------|----------|
| 技术负责人 | 自动化评审、代码质量把控 |
| 开发人员 | 快速获得反馈、学习改进 |
| DevOps | 安全合规、批量检查 |

## 2. 核心功能需求

### 2.1 多Agent并行评审

**功能描述**

系统内置多个专项Agent，同时对代码进行语法、风格、安全、性能四个维度的检查。每个Agent独立运行，最终汇总结果。

**交互逻辑**

1. 用户提交PR → 系统接收Webhook事件
2. 并行启动4个Agent进行检测
3. 各Agent返回结构化结果
4. 汇总生成统一评审报告

**验收标准**

- [ ] 支持至少4类Agent：语法检查、代码风格、安全扫描、性能建议
- [ ] 各Agent并行执行，总耗时不超过PR文件总行数×10ms（上限60秒）
- [ ] 结果格式统一，包含：文件路径、行号、问题类型、严重程度、修复建议

---

### 2.2 Git平台集成

**功能描述**

支持GitHub和GitLab两大主流平台，实现PR事件触发、评审结果回填、状态检查。

**交互逻辑**

1. 用户在GitHub/GitLab配置Webhook，指向评审服务
2. 收到PR事件（opened/reopened/synchronize）时自动触发评审
3. 评审完成后，自动在PR下发表评论
4. 更新PR状态（success/failure）

**验收标准**

- [ ] 支持GitHub Webhook配置
- [ ] 支持GitLab Webhook配置
- [ ] 评审结果自动以评论形式回填到PR
- [ ] 支持设置评审阈值，阻塞合并（可选）

---

### 2.3 多语言支持

**功能描述**

支持主流编程语言的代码评审，开箱即用。

**交互逻辑**

- 用户提交代码后，系统自动识别文件类型
- 根据语言类型调用对应的Agent进行检查

**验收标准**

- [ ] 支持JavaScript/TypeScript
- [ ] 支持Python
- [ ] 支持Java
- [ ] 支持Go
- [ ] 支持至少一种语言时，其他语言跳过（不报错）

---

### 2.4 配置化规则

**功能描述**

通过配置文件定义评审规则，支持团队自定义。

**交互逻辑**

1. 项目根目录放置 `.reviewer.yaml` 配置文件
2. 系统加载配置，覆盖默认规则
3. 根据配置执行评审

**验收标准**

- [ ] 支持配置启用/禁用特定Agent
- [ ] 支持配置严重程度阈值（Critical/Error/Warning/Info）
- [ ] 支持配置忽略特定文件或目录
- [ ] 配置格式为YAML，示例：

```yaml
agents:
  syntax: enabled
  style: enabled
  security: enabled
  performance: enabled
severity:
  block_on: error  # error及以上级别阻断合并
ignore:
  - "*.test.js"
  - "vendor/*"
```

---

### 2.5 评审报告展示

**功能描述**

提供清晰的评审结果展示，包括问题定位和修复建议。

**交互逻辑**

1. 评审完成后，生成结构化报告
2. 通过Webhook回调通知调用方
3. 支持API查询历史评审记录

**验收标准**

- [ ] 返回JSON格式报告，包含：问题数、严重程度分布、详细问题列表
- [ ] 每条问题包含：文件路径、行号、列号、类型、描述、建议修复方案
- [ ] 支持按严重程度排序

---

### 2.6 Webhook集成

**功能描述**

支持事件触发和外部通知。

**交互逻辑**

1. 系统提供Webhook端点（`/webhook/github`, `/webhook/gitlab`）
2. 外部系统发送事件到Webhook
3. 评审完成后，向配置的回调URL发送通知

**验收标准**

- [ ] 支持GitHub风格Webhook签名验证
- [ ] 支持自定义回调URL配置
- [ ] 回调内容包含：PR信息、评审摘要、报告链接

---

### 2.7 轻量部署

**功能描述**

支持Docker单容器部署，配置简单。

**交互逻辑**

1. 用户执行 `docker run` 启动服务
2. 通过环境变量或配置文件完成基础配置
3. 访问管理界面配置Git平台连接

**验收标准**

- [ ] 支持Docker部署，提供官方镜像
- [ ] 支持环境变量配置（端口、GitHub配置等）
- [ ] 首次启动时间不超过30秒
- [ ] 内存占用不超过512MB

---

### 2.8 RESTful API

**功能描述**

提供API接口供其他系统集成调用。

**验收标准**

- [ ] `POST /api/review` - 手动触发评审（传入代码内容）
- [ ] `GET /api/review/{id}` - 查询评审结果
- [ ] `GET /api/health` - 健康检查
- [ ] 支持Token认证

## 3. 非功能需求

### 3.1 性能

| 指标 | 要求 |
|------|------|
| 单次评审最大耗时 | ≤60秒（PR文件≤5000行） |
| 并发评审能力 | 支持≥3个PR同时评审 |
| API响应时间 | ≤200ms（不含评审耗时） |

### 3.2 兼容性

| 项目 | 要求 |
|------|------|
| 浏览器 | Chrome 90+、Edge 90+、Firefox 90+ |
| 部署环境 | Docker 20.10+、Linux（主流发行版） |
| Git平台 | GitHub全版本、GitLab 14+ |

### 3.3 易用性

| 指标 | 要求 |
|------|------|
| 首次部署文档 | 提供README，包含5步内完成部署 |
| 配置复杂度 | 基础配置≤5项 |
| 学习成本 | 用户5分钟内可完成首次评审 |

### 3.4 安全

- [ ] GitHub Webhook签名验证
- [ ] API Token认证
- [ ] 敏感信息加密存储
- [ ] 评审数据不外传（本地处理）
